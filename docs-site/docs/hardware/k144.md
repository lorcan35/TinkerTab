# K144 LLM module (M5Stack LLM Module Kit)

The escape hatch when Dragon is unreachable. Stack the K144 on top of Tab5 and you get on-device LLM inference + ASR + TTS, fully offline.

## Hardware topology — Mate carrier required

You need **three pieces** stacked in this order:

```
Tab5 base → Module13.2 LLM Mate carrier → K144 LLM Module
                    │
              USB-C powered
```

The Mate carrier is **not optional**. Stacking the K144 directly on Tab5 causes M5-Bus 5 V rail collisions that wedge the AX630C NPU silicon — the K144 will boot but inference hangs after ~30 seconds.

The Mate provides the K144's power via its USB-C input. Tab5 stays USB-C powered separately. Two cables, two power sources.

## What's inside the K144

- **NPU** — Axera AX630C, ~3.2 TOPS
- **CPU** — host Linux running on the AX630C SoC
- **Memory** — 4 GB RAM, 32 GB eMMC
- **Models pre-installed** — qwen2.5-0.5B (LLM), sherpa-ncnn streaming Zipformer (ASR), single_speaker_english_fast (TTS), KWS Zipformer (keyword spotting)
- **Wire** — UART_NUM_1 between Tab5 and K144 at 115200 8N1 (auto-bumps to 1.5 Mbps after warmup)
- **Tab5 pins** — TX = GPIO 6, RX = GPIO 7 (Port C UART, M5-Bus pins 16/15)

## Live performance

| Phase | Duration |
|------|----------|
| Cold-boot warmup | ~4.5 s (NPU model load on first probe) |
| Warm text turn | 2-3 s end-to-end |
| Text → ASR → LLM → TTS chain | streamed back; first audio in ~3 s |

That's *significantly* faster than Local mode on Dragon for short turns, because there's no network involved and the model is tiny.

## Voice modes that use it

- **`vmode=4` Onboard** — *every* turn goes to K144. Dragon never sees it. Selectable in Settings → Voice mode.
- **`vmode=0` Local with failover** — Tab5 routes to K144 only if Dragon WS is unreachable for ≥30 s AND K144 is warm. Toast: *"Using onboard LLM"*. When Dragon comes back, next turn returns to Dragon and shows *"Dragon reconnected"*.

## Health states (visible in Settings)

The K144 health chip in Settings has four states:

| State | Meaning |
|------|---------|
| 🟢 READY | Warm, model loaded, accepting inferences |
| 🟡 PROBING | Reset in progress or first-boot warmup |
| 🟠 NEEDS_ATTENTION | Auto-retry budget exhausted; tap to manually reset |
| 🔴 UNAVAILABLE | Software reset failed; needs power-cycle |

Tap the health chip to manually trigger a software reset (`POST /m5/reset` via the debug server). Live timing on hardware: ~9.6 s to recover.

## Live diagnostics

Settings → Voice → K144 also shows live `sys.hwinfo` data:

```
NPU 38.4°C · load 0 · v1.3
11 MODELS · 1 LLM · 2 ASR · 3 TTS · 2 KWS · 3 vision
```

That's `NPU temp`, `cpu_loadavg`, `firmware version`, and the model registry breakdown. Cached for 30 s; force-refresh with `POST /m5/refresh`.

## ADB debugging

The K144's *top* USB-C (M140 module port) exposes Axera ADB. From a dev host:

```bash
sudo adb kill-server && sudo adb start-server && sudo adb devices
sudo adb shell                                    # root login, no password
sudo adb forward tcp:10001 tcp:10001              # forward StackFlow JSON service
```

See [`docs/PLAN-k144-recovery.md`](https://github.com/lorcan35/TinkerTab/blob/main/docs/PLAN-k144-recovery.md) for the full StackFlow verb registry.

## Which K144 SKUs are supported?

We've validated against firmware **v1.3** (the version stamped on the Mate carrier model). Older firmwares may have a different model registry — `GET /m5/models` should still work, but the available capabilities differ.
