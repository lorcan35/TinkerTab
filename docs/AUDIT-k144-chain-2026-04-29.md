> **Source:** Three parallel audits (UI/UX, piping/data-flow, architecture) of commits `4ec33dc` + `3a6f67a` (Phase 6b chain APIs + chat overlay mic integration on PR #326).  Read-only; no code modified during audit.

## Headline

- **5 P0s** (functional correctness + UX cliffs that ship today)

- **5 things solid + worth preserving** as we refactor.

## UX-specific P0s

5. **Chat header chip displays "● Local" while in vmode=4.**  `chat_header_set_mode()` (`main/chat_header.c:215`) and `paint_header_and_view_for_mode()` (`main/ui_chat.c:166`) clamp `m > 3` to `0`.  `s_mode_short[4]` (`chat_header.c:28`) only has Local/Hybrid/Cloud/Claw.  Onboard mode is invisible in chat.
6. **Mode-cycle long-press never reaches vmode=4.**  `VOICE_MODE_COUNT=4` (`main/config.h:53`) wraps `0..3`.  Hardcoded `[4]` arrays at `chat_suggestions.c:52`, `ui_home.c:153/156`, `ui_chat.c:217/227`, `chat_header.c:25-28` all need to grow to 5.  Onboard is only reachable through Settings → Voice.

## Architecture-specific P0

7. **`voice.c` is 4,243 lines covering 5 modes + 7 lifecycles.**  Phase 4-6 added ~580 net lines.  `s_chain_*` state is split across two declarations 3,250 lines apart (a fragility the code itself comments on at `voice.c:3582-3585`).  Two more iterations at this pace make it unmaintainable.  Recommendation:  extract `main/voice_onboard.{c,h}`.

## P2 polish (follow-up backlog)

- Audit #1.1 — Chain dispatch is hard-coded to four units (OCP gap).  Adding kws/vad needs both struct field + `strcmp` branch.  `voice_m5_llm.c:1001-1007`.
- Audit #2.2 — Duplicate `s_chain_active` declarations 3,250 lines apart (`voice.c:326` and `:3581-3597`).  Resolved automatically by Wave 4's `voice_onboard` extract.
- UX #1 — Toasts during chain start/stop are auto-dismissing; users can blink and miss them during the 5-sec setup.  Consider non-toast indicator.
- UX #4 — Setup-failed toast is generic ("Onboard chain unavailable") — not actionable.  Map common error codes to specific toasts.
- UX #5 — 10-min hard cap is silent and absolute.  Toast should distinguish "you stopped" from "session limit reached".
- Piping #2 — Frame consumption uses `memmove(s_rx_buf, s_rx_buf + consumed, ...)` — O(N²) under load.  Negligible at 115200 baud, worth revisiting if Phase 6a's 1.5 Mbps gets wired into chain path.
- Piping #2 — Oversized-frame wedge:  if a single TTS frame exceeds 384 KB, the recv loop deadlocks until the 600 s timeout.  Current K144 models don't produce frames this large; defensive guard is cheap.
- Piping #6 — State-machine race:  WS reconnect handler can clobber chain's `LISTENING` state with `RECONNECTING` mid-chain.  Visible only as a state-badge flicker.
- Piping #7 — LLM accumulator silently truncates past 2 KB; no log on overflow.
- Piping #7 — TTS b64 overflow leaves partial bytes in buffer; next utterance decodes them as a quartet-misaligned prefix → garbled audio for one utterance.
- Piping #8 — Chain runs at default 115200 baud; never bumps to the verified 1 Mbps.  ~4× slower TTS payload transfer than necessary.
- Arch #5 — Long-lived task spawned via `xTaskCreate` instead of the worker queue (correct call but worth a comment in CLAUDE.md "Concurrency primitives" section).
- Arch #8.2 — `voice_m5_llm.c` mixes sync/caller-buffered API (`voice_m5_llm_tts`) with async-callback API (`voice_m5_llm_chain_run`) in the same file.  Defensible but document the idiom split in the file header.
- Arch #9.1 — Plan doc claims Phase 6b "DONE" but the documented Eigen workaround is not yet applied.  Honest status:  "DONE for ASR + LLM; TTS via chain BLOCKED by vendor; workaround PENDING".

## What the audits agreed is solid

1. **Opaque chain handle.**  `typedef struct voice_m5_chain_handle voice_m5_chain_handle_t` in the header; the struct body lives in `.c`.  Textbook DIP.  Worth preserving in the refactor.
2. **`chain_setup_unit` skip-non-matching-frames pattern.**  Caught a real protocol gotcha (upstream stream frames arriving while waiting for downstream ACK).  Code comment + LEARNINGS entry both reference it.
3. **PSRAM-lazy buffer alloc.**  The boot-loop fix (BSS overflow → `vApplicationGetTimerTaskMemory`) is the canonical pattern, same as `ui_sessions.c`.
4. **Drain task spawn (not worker-queue).**  Correct primitive choice for a 10-min consumer; the comment block documenting *why* (`voice.c:3744-3751`) will save the next reader an hour.
5. **`VMODE_LOCAL_ONBOARD` short-circuit ordering in `voice_start_listening`.**  Placing the K144 branch BEFORE `mic_mute` / `ws_live` checks is right (those are about Tab5's mic + Dragon WS, neither relevant in chain mode).  Easy to get wrong.
6. **Cumulative-vs-additive ASR/LLM accumulator semantics.**  sherpa-ncnn's full-replace partials vs the LLM's token concat.  Easy to get backwards; got right.
7. **Reverse-order teardown** (`tts → llm → asr → audio`).  Each downstream subscriber dies before its publisher, avoiding K144-side `Stream data index error -25`.
8. **Memory paths in `chain_run` are leak-free under all 4 alloc-fail branches and the success path.**  Hard to get right under `goto`-less C.

## Top 3 architectural moves (recommended order)

1. **Port-C UART mutex** at the transport layer (`bsp/tab5/uart_port_c`).  Closes the silent UART-collision class permanently.  Pre-requisite for everything else.  Closes #1, #3, partially #4.  ~50 LOC.
2. **TTS Eigen workaround.**  Strip `tts.setup` from `voice_m5_llm_chain_setup`; call `voice_m5_llm_tts` from `chain_text_callback` on LLM `finish=true`.  Combined with the mutex, this makes TTS through Tab5's speaker actually work.  Closes #16; unblocks the headline UX promise.
3. **Extract `main/voice_onboard.{c,h}`** + the [4]-arrays/clamp-removal/`TH_MODE_ONBOARD` PR.  Decoupled work, can run in parallel with #1/#2.  Stops voice.c from accreting the next phase too.  Closes #5, #6, #7.
