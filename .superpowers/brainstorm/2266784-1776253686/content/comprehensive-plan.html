3. Tab5 Firmware Changes

3.1 Enable JPEG Decoder

Option A (recommended): LVGL software TJPGD

Add to sdkconfig.defaults: CONFIG_LV_USE_TJPGD=y

Tiny JPEG Decoder — ~8KB code, decodes JPEG → RGB565 in PSRAM. Slow (~200ms for 660x400) but zero dependencies. Good enough for inline chat images.

Option B (future): Hardware JPEG decoder

ESP32-P4 has a dedicated JPEG decoder IP. Requires esp_jpeg component (add to idf_component.yml). ~10x faster. But needs custom LVGL decoder registration. Save for v2.

Decision: Start with TJPGD (software). It's one line in sdkconfig. Hardware decoder is a performance optimization for later.

3.4 ui_chat.c Changes (~200 lines)

Three new rendering functions alongside existing ui_chat_add_message():

ui_chat_push_media() — lv_async_call wrapper → add_image_bubble()

  • Creates container at s_next_y (same pattern as text bubbles)
  • Shows "Loading..." placeholder (1 label object)
  • Kicks off media_cache_fetch() on a background task
  • On completion: lv_async_call → replace placeholder with lv_image (2 objects: container + image)
  • Tap handler: fullscreen image zoom (reuse existing overlay pattern)

ui_chat_push_card() — lv_async_call wrapper → add_card_bubble()

  • 4 objects: container + title label + subtitle label + optional thumbnail
  • Dark background card with rounded corners (matches existing chat style)
  • Max width: 460px (same as text bubbles)

ui_chat_push_audio_clip() — lv_async_call wrapper → add_audio_bubble()

  • 3 objects: container + play button + duration label
  • Tap play → download audio → play via existing I2S pipeline
  • One clip at a time (stop current before starting new)
Object budget impact: With 30 messages, current max is ~90 objects. Each image adds 2, each card adds 4, each audio adds 3. Worst case (all 30 messages are images): 60 objects for images alone. Mitigation: Reduce max messages from 30 to 20 when media is present, or implement lazy loading (only render visible messages).

3.5 Camera-to-Chat (ui_camera.c, ~50 lines)

Add "Send to Chat" button on camera screen. Flow:

  1. User captures photo (existing RGB565 frame)
  2. Encode as JPEG via TJPGD encoder (or send as BMP to Dragon, let Dragon convert)
  3. HTTP POST to Dragon /api/media/upload → gets media_id
  4. Send WS message: {"type":"user_media","media_id":"...","text":"..."}
  5. Navigate to Chat, show user's photo as a bubble
  6. Dragon forwards image to multimodal LLM (Claude 3.5 Sonnet, GPT-4o)
JPEG encoding: TJPGD is decode-only. For encoding, we can either: (a) send raw BMP to Dragon and let Dragon convert, (b) add esp_jpeg encoder component. Option (a) is simpler — BMP is larger (~1.8MB for 720p) but Dragon is on the same LAN, transfer is fast.

4.4 server.py Changes (~60 lines)

Intercept LLM response stream, detect media opportunities, interleave media events.

Key change: Instead of sending each token immediately, buffer complete sentences, check for media patterns, then send text + media events.

# Current (server.py line ~1391):
await ws.send_json({"type": "llm", "text": token})

# New: Buffer tokens, detect media at sentence boundaries
sentence_buf += token
if sentence_complete(sentence_buf):
    media_items = await media_pipeline.detect_and_render(sentence_buf)
    for item in media_items:
        if item["type"] == "text":
            await ws.send_json({"type": "llm", "text": item["content"]})
        else:
            await ws.send_json(item)  # media, card, audio_clip
    sentence_buf = ""
Latency concern: Buffering by sentence adds latency before first token appears. Mitigation: Stream tokens immediately as before. Run media detection AFTER llm_done on the full response. Send media events after the text is complete. User sees text first, then images load in — like a web page. No perceived latency increase.

7. Risk Assessment

RISK 1: Heap fragmentation from image alloc/free

Image pixel buffers (581KB each) are allocated/freed from PSRAM as cache evicts. Repeated alloc/free of large blocks fragments PSRAM.

Mitigation: Pre-allocate 5 fixed-size PSRAM slots at boot. Cache fills slots, never frees them — just overwrites with new image data. Zero fragmentation. Cost: ~3MB permanent PSRAM reservation.

RISK 2: HTTP download stalls voice WS

If image download saturates WiFi, voice WebSocket packets could be delayed or dropped.

Mitigation: Download in 4KB chunks with 10ms yield between chunks. Download task at priority 2 (below voice at 4-6). If voice is active (LISTENING/PROCESSING/SPEAKING), pause downloads entirely.

RISK 3: LLM response with 10 image URLs

AI returns a response with many embedded URLs. Dragon tries to render all of them. Tab5 gets flooded with media events.

Mitigation: Cap at 3 media items per response. Dragon renders first 3 URLs, remainder sent as text links only. Tab5 shows max 3 images per message.

RISK 4: Dragon media store fills disk

Rendered images accumulate in /home/radxa/media/.

Mitigation: Auto-cleanup: delete files older than 24 hours. Max directory size: 500MB. Cleanup runs hourly as asyncio task.

LOW RISK: Protocol backward compatibility

All new message types are additive. Old firmware ignores them. New firmware degrades gracefully to text-only if Dragon doesn't send media. Version negotiation via register message's firmware_ver field.

8. Implementation Order (4 Phases)

Phase 1: Foundation (Dragon renders, Tab5 displays)

Deliverable: Images from LLM responses appear inline in chat

  1. Enable TJPGD in sdkconfig.defaults (1 line)
  2. Create media_cache.c on Tab5 (HTTP download + JPEG decode + PSRAM cache)
  3. Add media/card/audio_clip message types to voice.c dispatcher
  4. Add image bubble renderer to ui_chat.c
  5. Create media_store.py + media_routes.py on Dragon (serve + upload)
  6. Create media_pipeline.py on Dragon (image URL detection + proxy + resize)
  7. Wire media detection after llm_done in server.py
  8. Test: ask "Show me a picture of a cat" → image appears inline

~800 new lines across both repos. ~3-4 days.

Phase 2: Code & Table Rendering

Deliverable: Code blocks and tables render as styled images

  1. Add Pygments to Dragon requirements (syntax highlighting)
  2. Code block renderer in media_pipeline.py (detect ```...``` → Pillow render with syntax colors)
  3. Table renderer (detect markdown tables → Pillow render as styled grid)
  4. Markdown basic formatting (bold, headers, lists → styled image)
  5. Test: ask for code or comparisons → rich renders inline

~300 new lines on Dragon. ~2 days.

Phase 3: Camera-to-Chat (User sends media)

Deliverable: User photographs something, AI analyzes it

  1. Add "Send to Chat" button on camera screen
  2. HTTP upload flow (BMP → Dragon → convert to JPEG)
  3. user_media WS message type in voice.c
  4. Dragon: load image, encode base64, forward to multimodal LLM
  5. Mode check: only Cloud + TinkerClaw support vision. Show error in Local mode.
  6. Test: photograph something → ask "What is this?" → AI describes it

~200 new lines Tab5, ~100 Dragon. ~2 days.

Phase 4: Rich Cards & Web Previews

Deliverable: Link previews, audio clips, search result cards

  2. Audio clip player in ui_chat.c (download + I2S playback)

  5. Search result cards from TinkerClaw web_search (extract + render)
  6. Test: paste URL → link preview card. Ask for music → audio preview plays.

~400 new lines across repos. ~3 days.
