TinkerClaw Mode UI: Informed by Full Codebase Analysis

After reading every file across TinkerTab, TinkerBox, and TinkerClaw. Here's what's actually possible given the architecture.

Key Discovery: Tool Events Are Invisible

TinkerClaw's /v1/chat/completions SSE stream only sends text deltas. When the agent calls web_search or memory_recall, the stream goes silent for 10-30s. Dragon's tinkerclaw_llm.py (192 lines) has no idea what tool is running. Tab5 sees nothing.

However, TinkerClaw does emit rich tool events internally (stream: "tool" with phase/name/args/result). They just aren't in the SSE response Dragon consumes.

A

Smart Chat (Silence Detection + Inline Indicators)

No new screen. Chat gets agent-aware enhancements using what we CAN detect today:

  • Silence detection: If no LLM token arrives for 3s during PROCESSING, show animated "Agent working..." bubble (Dragon can detect this in tinkerclaw_llm.py and send a new WS event)
  • TinkerClaw suggestion cards: "What do you know about me?", "Search the web for...", "What can you do?", "Remember that I..."
  • Fix model label: Show "TinkerClaw Agent" instead of "Claude Haiku"
  • Agent welcome card: On chat home, show capabilities: "I can search the web, remember things, use tools"
  • Rose theme accents: Bubbles, status dots, mode badge all use TinkerClaw rose color
Changes: ui_chat.c (suggestion cards, model label, welcome card, silence indicator) + tinkerclaw_llm.py (silence detection event) + voice.c (forward new event to UI)
Objects: +5-6 | Risk: Low | No new screens or files
B

Full Tool Visibility (Parallel Event Channel)

Add a WebSocket connection from Dragon to TinkerClaw's event system to get real tool names, args, and results. Then forward them to Tab5 as tool_call/tool_result messages (which voice.c already handles!).

  • Dragon subscribes to TinkerClaw agent events via WebSocket (new connection alongside HTTP)
  • Receives: stream:"tool", phase:"start", name:"web_search", args:{query:"..."}
  • Forwards: {"type":"tool_call","tool":"web_search","args":{...}} to Tab5 (existing protocol!)
  • Tab5 already handles tool_call/tool_result in voice.c → shows "Searching the web..." in ui_voice.c
  • Plus everything from Option A (suggestions, model label, welcome card)
Changes: tinkerclaw_llm.py (WS event subscription, ~80 lines) + server.py (forward tool events) + ui_chat.c (cosmetic) + ui_voice.c (already handles tool_call!)
Objects: +5-6 | Risk: Medium — new WS dependency, but Tab5 side already done
Huge win: Tab5 voice.c already shows "Searching...", "Calculating..." for tool_call events
C

Inline SSE Tool Injection (Modify TinkerClaw)

Modify TinkerClaw's OpenAI HTTP endpoint to inject tool events INLINE in the SSE stream. Dragon parses them as special SSE events alongside text deltas. Simplest server-side change.

  • Modify openai-http.ts to write tool events into SSE stream: data: {"tool":"web_search","phase":"start"}
  • Dragon parses in tinkerclaw_llm.py alongside text deltas (already loops over SSE lines)
  • Forwards as tool_call/tool_result to Tab5 (same as Option B)
  • No new WebSocket connections — everything stays in the existing HTTP SSE stream
  • Plus everything from Option A
Changes: openai-http.ts (inject tool SSE events, ~30 lines) + tinkerclaw_llm.py (parse tool events, ~40 lines) + server.py (forward) + ui_chat.c (cosmetic)
Objects: +5-6 | Risk: Low-Med — modifying OpenClaw source, but minimal change
Cleanest: Single connection, no new WebSocket, all in existing data flow
D

Voice Orb Agent Cockpit

The voice overlay (ui_voice.c) transforms when in TinkerClaw mode. The orb becomes an agent status display. Tool activity shown as animated rings/text around the orb. Chat stays for text history.

  • Orb color ring: Rose pulsing = thinking, amber flash = tool executing, green = speaking
  • Status label evolution: "Listening..." → "Thinking..." → "Searching the web..." → "Found 3 results" → "Speaking..."
  • Context ring: Thin ring around orb showing memory/tool activity count (like Apple Watch activity rings)
  • Tool name display: Below orb, shows current tool: "web_search" with animated dots
  • Requires tool visibility from Option B or C to show real tool names
Changes: ui_voice.c (orb state machine extensions, ~100 lines) + needs B or C for tool data
Objects: +4-6 (reuses existing orb objects) | Risk: Low — ui_voice.c already has the state machine
Voice-first: Most interactions are via orb, so this is where visibility matters most
E

Combined: C + A + D (The Full Package)

The recommended approach. Three layers that build on each other:

  1. Layer 1 — SSE tool injection (C): Modify TinkerClaw's openai-http.ts to emit tool events in SSE. Dragon parses and forwards to Tab5. ~70 lines across 2 files.
  2. Layer 2 — Smart Chat (A): Agent suggestions, model label fix, welcome card, tool activity bubbles inline. All in existing ui_chat.c.
  3. Layer 3 — Voice Orb Cockpit (D): Orb shows tool names, animated status, rose theming. Uses tool events from Layer 1. All in existing ui_voice.c.
Changes: openai-http.ts + tinkerclaw_llm.py + server.py + ui_chat.c + ui_voice.c (all existing files)
Objects: +10-12 total | Risk: Medium — touches 3 repos but each change is small
Zero new files. Zero new screens. Zero new LVGL overlays.
F

Agent Dashboard Screen (New Dedicated Screen)

A brand new screen (ui_agent.c) accessible from nav bar when in TinkerClaw mode. Shows: conversation feed, tool history, memory facts, skill list, agent config. Full mission control.

  • Conversation panel: Recent messages with tool indicators
  • Tool activity feed: Timestamped log of tool executions
  • Memory viewer: "What the agent knows about you"
  • Skill browser: Scrollable list of 51 skills
  • Config panel: Model, token budget, context usage
Changes: New ui_agent.c (~1500 lines) + ui_home.c (nav bar mod) + needs B or C for data
Objects: +30-40 (DANGER: pushes 55-object practical limit per screen) | Risk: HIGH
Problem: 720px too narrow for panels. 55 objects per screen budget is tight. New overlay = fragmentation risk.

My Recommendation: E (Combined C+A+D)

Why: It gives you real tool visibility (not heuristic silence detection), surfaces it in both chat AND voice (the two places users actually interact), modifies zero LVGL screen architecture, adds zero new files to Tab5, stays within object budget, and each layer can be implemented and tested independently.

Implementation order: C first (enable the data flow), then A (chat cosmetics), then D (voice orb). Each layer is independently useful — you don't need all three to ship.

What NOT to do: Option F (new screen) fights the hardware. 720px is too narrow for panels, 55 objects per screen is too tight for a dashboard, and a new overlay risks fragmentation. The existing chat + voice screens are the right containers.
