Rich Media Architecture: How To Do All Of It

One architecture that enables all 120 user stories. Dragon renders, Tab5 displays.

The Core Insight

Tab5 can do three things: show images, play audio, show text. Dragon can do anything. So: Dragon renders everything into images/audio, Tab5 displays them inline. One pattern, 120 stories.

Layer 1: Rich Message Protocol

New WS message types: media, card, audio_clip, carousel. Dragon REST API for media serving.

Protocol (both)

Layer 1: The Protocol Change

Current: Dragon sends only text

{"type":"llm","text":"The weather in Dublin is 14°C..."}
{"type":"llm_done","llm_ms":1523}

New: Dragon sends text + media interleaved

{"type":"llm","text":"Here's the weather in Dublin:"}
{"type":"media","media_type":"image","url":"/api/media/weather_abc.jpg","width":680,"height":400,"alt":"5-day forecast"}
{"type":"llm","text":"14°C with light rain expected."}
{"type":"media","media_type":"card","title":"Dublin Weather","subtitle":"14°C · Light Rain","image_url":"/api/media/weather_icon.png","action":"expand"}
{"type":"llm_done","llm_ms":2100}

New message types:

media — inline image/audio/video-frame: url, media_type, width, height, alt
card — rich preview card: title, subtitle, image_url, description, action
carousel — swipeable image set: items[] (each with url, caption)
audio_clip — inline audio player: url, duration_s, label

Layer 2: Tab5 Media Renderer

Tab5 receives a media message → downloads image from Dragon's HTTP API → decodes JPEG in PSRAM → creates lv_image in chat scroll area.

Image Display
  • HTTP GET from Dragon API
  • JPEG decode → RGB565 in PSRAM
  • Max width: 660px (bubble width)
  • lv_image with rounded corners
  • Tap to fullscreen zoom
Memory Management
  • LRU cache: 5 images in PSRAM
  • ~660×400 RGB565 = ~528KB each
  • 5 cached = ~2.6MB (of 32MB PSRAM)
  • Offscreen images → placeholder
  • Re-download on scroll back
Card Renderer
  • Title + subtitle + thumbnail
  • ~4 LVGL objects per card
  • Rounded container, dark bg
  • Tap action: expand / open URL
  • Reuses existing chat bubble style
Audio Player
  • Mini player widget in chat
  • Play/pause button + duration
  • Download + I2S playback (existing)
  • One at a time (stop current first)
  • ~3 LVGL objects

Layer 3: Dragon Media Pipeline

Dragon renders anything into 720px-wide images. The rendering engine is a Python module with pluggable renderers.

1
Playwright Renderer

Web screenshots, HTML→image, link previews, page captures. Already installed on Dragon. Headless Chrome renders any HTML/CSS at 720px viewport → PNG screenshot → JPEG optimize.

2
Pillow Renderer

Charts (matplotlib), QR codes (qrcode), image composites, annotated photos, text-on-image. Pure Python, fast, no browser needed.

3
Markdown/Code Renderer

Markdown → HTML → Playwright screenshot. Code blocks → syntax-highlighted HTML → screenshot. Tables → styled HTML → screenshot. All use the same Playwright pipeline.

4
Media Proxy

External image URLs → download → resize to 720px max → optimize JPEG quality 80 → serve locally. Tab5 never fetches from the internet directly — Dragon proxies everything.

5
Media Store

All rendered media saved to /home/radxa/media/ with unique IDs. Served via GET /api/media/{id}.jpg. Auto-cleanup after 24h. Session-scoped for privacy.

Layer 4: User Media Input (Camera/Audio → AI)

Camera → Chat

Tab5 camera captures JPEG (already works in ui_camera.c) → user taps "Send to Chat" → HTTP POST to Dragon /api/media/upload → Dragon stores + forwards as base64 to multimodal LLM (Claude 3.5/GPT-4o accept images) → response streams back normally.

Audio → Chat

Already works for voice (PTT). For audio clips: user records short audio → upload to Dragon → Dragon transcribes OR sends to audio-capable model (Whisper/GPT-audio) → response includes analysis.

New WS message: user_media
{"type":"user_media","media_type":"image","media_id":"upload_abc123","text":"What plant is this?"}

Tab5 uploads image first via HTTP, gets media_id, then sends WS message with media_id + optional text context.

Layer 5: AI Media Detection

Dragon intercepts LLM responses and auto-renders rich content. The LLM doesn't need to know about rendering — Dragon detects patterns and acts.

URL detection — LLM response contains https://... → Dragon fetches Open Graph metadata → sends as card message
Image URL detection — Response contains .jpg/.png/.gif URL → Dragon downloads, resizes → sends as media message
Code block detection — Response contains ```python...``` → Dragon renders with syntax highlighting → sends as media image
Table detection — Response contains | header | header | pattern → Dragon renders as styled table image
Markdown detection — Response contains headers, lists, bold → Dragon renders formatted version as image
TinkerClaw tool results — Web search returns URLs/images → Dragon renders search results as cards

End-to-End Flow Example

User: "What does a Japanese maple look like in autumn?"

1. Tab5 → Dragon: {"type":"text","content":"What does a Japanese maple look like in autumn?"}
2. Dragon → LLM: sends to Claude/TinkerClaw
3. LLM responds: "Japanese maples (Acer palmatum) are stunning in autumn... ![maple](https://upload.wikimedia.org/...maple.jpg)"
4. Dragon detects image URL → downloads → resizes to 660px → saves as /api/media/maple_001.jpg
5. Dragon → Tab5: {"type":"llm","text":"Japanese maples are stunning in autumn..."}
6. Dragon → Tab5: {"type":"media","media_type":"image","url":"/api/media/maple_001.jpg","width":660,"height":440}
7. Tab5 renders text bubble, then downloads image, then shows inline below text. User sees beautiful maple photo in chat.

Why This Works

  • One pattern for everything: Charts, web pages, code, diagrams, tables, PDFs — all become images. Tab5 only needs one renderer (lv_image).
  • Tab5 stays thin: No HTML engine, no markdown parser, no video decoder on the ESP32. Just image display + audio playback.
  • Dragon has unlimited power: Playwright, Pillow, matplotlib, ffmpeg — full Linux arsenal for rendering.
  • Tab5 never touches the internet: Dragon proxies all external media. Privacy preserved. No DNS/TLS on ESP32.
  • Backward compatible: Old firmware ignores unknown message types. New firmware handles both text-only and rich responses.
  • Mode-agnostic: Works for Local, Hybrid, Cloud, AND TinkerClaw. Dragon renders for all modes.
  • Zero TinkerClaw changes: Dragon intercepts and renders. TinkerClaw just returns text.
